\section*{摘要}
\addcontentsline{toc}{section}{摘要}

面向科学发现的符号回归研究，正在同时经历两条智能化演进路径。第一条路径是 AI4Science：原本由人类科学家观察数据、归纳规律并总结公式的过程，正在被符号回归方法转化为“机器从数据中发现数学表达式”的过程。第二条路径是 AI for AI4Science：原本由人类算法研究者手工设计、调试和改进符号回归算法的过程，正在被大语言模型与 Agent 系统转化为“机器自动改进科学发现工具”的过程。与这一趋势相比，当前研究仍存在两个突出短板：其一，符号回归方法生态高度碎片化，不同仓库在环境依赖、数据格式、执行流程和评测口径上缺乏统一规范；其二，自动化改进方法虽然开始出现，但大多缺少清晰的协议边界、审查机制与可复现平台支撑。

基于上述问题，本文拟开展《面向科学发现的符号回归评测平台与自动化算法改进方法研究》。本课题的第一层目标，是构建面向科学发现的统一符号回归评测平台，重点解决异构工具接入、规范化数据集模式、协议编译、隔离执行、结果归档以及多维评测等问题；第二层目标，是在统一平台之上研究审查驱动的自动化算法改进方法，重点解决平台状态表示、受约束动作空间、多目标收益函数以及协议门控等问题，使 LLM Agent 的改进行为能够在不破坏 benchmark 语义和真值隔离原则的前提下有边界地开展。

本课题具有较好的现实基础。当前项目原型已集成 $10$ 个异构符号回归后端、$8$ 个受控执行环境、多套工具验收脚本以及一批可用于公式验证与 benchmark 试点的数据资产，说明研究对象真实存在，平台问题已经暴露，自动化改进也具备直接落地的工程起点。本文将综合采用文献调研、系统设计、协议建模、实验验证、案例分析与约束优化等方法，预期形成一套可复现的符号回归评测平台、一套审查优先的自动改进方法、配套的实验方案与评测基线，并为后续学位论文与系统扩展提供稳定研究基础。

\noindent\textbf{关键词：}符号回归；科学发现；评测平台；自动化算法改进；大语言模型 Agent；AI4Science

\section{研究背景与选题意义}

\subsection{研究背景}

科学研究的一个核心任务，是从观测数据中识别潜在规律，并进一步将这些规律表达为可解释、可验证、可推广的数学形式。在传统范式下，这一过程通常由人类科学家完成：研究者基于经验提出假设，通过实验采集数据，再借助建模、推理与验证逐步形成公式、定律和机制解释。随着数据规模、系统复杂度和跨学科耦合程度持续上升，单纯依赖人工经验进行规律归纳正在面临明显的效率瓶颈。一方面，现代科学实验与仿真能够产生海量高维数据；另一方面，真正有价值的科学规律往往需要同时满足精度、稀疏性、可解释性与外推性，这使得“从数据到公式”的问题成为 AI4Science 中最具代表性的任务之一。

符号回归（Symbolic Regression, SR）正是在这一背景下逐步成为面向科学发现的重要方法。与一般的黑盒预测模型不同，符号回归并不只追求训练集上的拟合精度，而是试图直接生成人类可读的数学表达式。自 \citet{schmidt2009distilling} 提出通过机器从实验数据中“蒸馏”自然规律以来，符号回归逐步发展为一个横跨遗传编程、组合搜索、神经引导生成、预训练模型和大语言模型推理的研究方向。AI Feynman 系列工作 \citep{udrescu2020aifeynman,udrescu2020aifeynman2} 强调了物理启发式分解与模块化搜索的价值；深度符号回归及其统一框架 \citep{petersen2021deep,landajuela2022udsr} 证明了神经策略与风险偏好优化在公式搜索中的可行性；PySR 等工程化工具 \citep{cranmer2023pysr} 则提升了符号回归在科研与工业环境中的可用性。这说明机器已经不再只是帮助科学家做数值拟合，而是开始参与到“发现公式”这一更高层级的科学推演过程中。

然而，当符号回归真正被用于科学发现时，研究者会迅速遇到另一个更偏工程的问题：现有 SR 生态极其碎片化。不同方法通常以独立仓库形式发布，各自依赖不同的运行时环境、算子定义、数据格式、执行脚本和输出规范。即便使用同一数据集，不同仓库往往也会采用不一致的训练/验证/测试划分方式，对表达式复杂度、运行预算、超参数搜索、常数优化和 OOD 评测的处理更是差异显著。这意味着很多研究工作在真正比较算法能力之前，就已经被环境配置、数据重构和协议对齐所消耗。

在此基础上，本文关注的第二个背景变化更加值得重视。传统的符号回归算法研究，主要依赖人类计算机学者提出新搜索策略、设计算子、调试执行框架并手工做实验对比；但随着大语言模型和 Agent 系统的发展，算法本身的设计与改进也开始进入自动化阶段。FunSearch 通过大语言模型结合程序搜索来发现新的数学构造 \citep{romeraparedes2024funsearch}，AlphaDev 展示了强化学习自动发现更快排序算法的能力 \citep{mankowitz2023alphadev}，The AI Scientist 与 AutoSOTA 等工作进一步探索了从实验生成到研究闭环的自动化流程 \citep{lu2024aiscientist,li2026autosota}。由此可见，AI4Science 领域正在形成新的二层结构：第一层是机器代替人类科学家从数据中发现规律；第二层则是机器进一步代替人类算法研究者去设计、改进和演化发现规律的工具。

本课题正是建立在这两个层次之上展开的。第一层问题是：如何构建一个真正面向科学发现的符号回归评测平台，使异构算法、异构数据和异构实验协议能够在同一研究基底上被运行和比较。第二层问题是：在这一统一平台之上，如何引入 LLM Agent 对符号回归算法及其配置进行自动化改进，同时确保改进过程不破坏评测公平性，不泄露真值信息，不引入不可追溯的协议漂移。相比只做单一算法精度优化，这一选题更直接地面向“科学发现流程的智能化基础设施”。

\subsection{选题意义}

从理论层面看，本课题的意义首先在于为“科学发现中的符号回归”提供一个更清晰的系统研究视角。当前许多工作将注意力集中在单个模型、单类搜索策略或单个 benchmark 上，但对平台、协议与自动改进之间的关系关注不足。事实上，符号回归要真正服务科学发现，不仅要能找到公式，还要能在严格、稳定、可复现的研究流程中找到公式。也就是说，算法能力、评测纪律和自动化机制应被放在同一研究对象中讨论。本文试图把这三者统一起来，从而把“算法研究”推进为“研究基础设施与元算法协同研究”。

从方法论层面看，本课题有助于回答一个更广义的问题：当 AI 系统开始承担越来越多的科研功能时，我们如何为其提供可验证、可审计、可追溯的操作边界。若没有统一平台，自动化算法改进很容易退化为对局部指标的无约束追逐，甚至通过修改数据划分、利用真值公式、切换评测脚本等不公平方式获得表面性能提升。因此，研究“审查优先、评测驱动、边界清晰”的自动改进机制，不仅对符号回归有意义，也对更广泛的自动化科研系统具有方法论参考价值。

从应用层面看，本课题能够直接降低符号回归研究的工程门槛。一个统一的评测平台可以使新方法更快接入，使新数据集更快标准化，使对比实验更容易复现；而自动化改进机制可以帮助研究者在受控范围内快速定位失败模式、生成候选修改方案并筛选可行方向，从而显著提高科研迭代效率。特别是在跨学科场景中，领域科学家往往更关注公式质量和科学解释，而不具备处理复杂仓库与环境问题的经验，因此平台化与自动化的价值会更加明显。

从学科交叉层面看，本课题横跨符号回归、科学机器学习、软件工程、实验协议设计、自动化机器学习与大语言模型 Agent。其研究成果既可以表现为软件平台与实验规范，也可以表现为受约束自动改进算法和分析方法。这样的研究结构有利于形成可持续扩展的研究路线：短期可支撑硕士阶段开题、中期检查和论文撰写，长期则可扩展为更大规模的 AI for Science 基础设施研究。

\subsection{研究定位}

需要强调的是，本课题的核心目标不是再提出一个孤立的符号回归新模型，也不是仅仅实现一个工具箱式的软件集合。本文更关心两个层面之间的连接关系。第一层，平台必须能够真实承载科学发现型符号回归任务，包括公式恢复、OOD 泛化、表达式复杂度控制和真值验证等；第二层，自动化改进必须建立在平台输出的结构化证据之上，而不是脱离评测上下文的盲目搜索。因此，本文的研究定位可以概括为：\textbf{研究符号回归科学发现任务的统一评测基底，并在该基底上研究审查驱动的自动化算法改进方法。}

\section{国内外研究现状}

\noindent 从现有公开研究看，围绕本课题已经逐步形成四条彼此相关但尚未真正打通的研究线索：第一条是面向科学发现的符号回归方法研究，核心问题是“如何更好地从数据恢复公式”；第二条是 benchmark、数据集与评测协议研究，核心问题是“如何更公平地比较方法”；第三条是可复现平台与研究基础设施研究，核心问题是“如何让实验可运行、可复查、可扩展”；第四条是自动算法发现与 Agent 化研究，核心问题是“如何让机器自动提出并验证改进动作”。已有工作分别在这些方向上取得了重要进展，但尚缺少一个同时覆盖统一评测平台与自动化算法改进的整体研究框架。基于这一判断，本文将现有研究归纳如下。

\subsection{经典符号回归与科学发现}

符号回归长期以来都被视为连接数据驱动学习与科学规律表达的重要桥梁。早期代表性工作 \citet{schmidt2009distilling} 证明了机器可以直接从原始实验数据中恢复具有科学意义的函数关系，这使“机器参与科学发现”从概念设想变成了可验证的研究方向。随后，AI Feynman 系列工作 \citep{udrescu2020aifeynman,udrescu2020aifeynman2} 将变量分离、物理启发式假设和分而治之策略引入搜索过程，进一步强调了符号回归在科学规律发现中的特殊性，即它面对的不是一般意义上的数值预测，而是结构化、可解释、可外推的规律恢复问题。

这一路线的核心贡献，在于把符号回归从“表达式拟合技术”提升为“科学规律发现工具”。但从研究现状看，其主要局限也同样明显：一方面，许多工作更关注单个方法在特定任务上的成功案例，而较少讨论跨方法、跨平台的统一比较；另一方面，方法的有效性往往与先验设计、算子约束、变量分解和任务假设深度耦合，使得不同论文中的结果难以直接横向对比。对本课题而言，这一方向最重要的启发是：科学发现语境下的符号回归必须同时考察精度、复杂度、真值一致性和外推能力，因此平台层不能只提供“跑通算法”的能力，还必须提供支撑科学比较的协议与指标体系。

\subsection{深度符号回归、预训练模型与统一方法家族}

随着深度学习的发展，符号回归逐步从传统遗传编程扩展到神经引导生成、强化学习搜索和大规模预训练等路线。Deep Symbolic Regression \citep{petersen2021deep} 展示了风险偏好策略梯度在表达式空间搜索中的潜力；神经引导的种群初始化方法 \citep{mundhenk2021seeding} 将神经模型与遗传编程演化结合；Neural Symbolic Regression that Scales \citep{biggio2021nsr} 则强调了预训练与样本效率；Transformer 路线上的 SymbolicGPT、End-to-end Symbolic Regression 和 SymFormer \citep{valipour2021symbolicgpt,kamienny2022end,vastl2022symformer} 进一步将公式恢复表述为序列建模任务。总体来看，深度符号回归研究显著提升了方法表达能力，也把“公式搜索”从启发式演化拓展为可学习的建模问题。

在这一背景下，uDSR \citep{landajuela2022udsr} 的意义尤其值得注意。它并非单纯追求更高分数，而是尝试在同一框架内统一多种深度符号回归变体，使不同设计能够在共享实现基底上被分析和比较。这说明“统一化”本身已经成为方法研究的一部分。不过，现有统一框架主要针对同一方法家族内部的模块组合问题，尚未触及更加棘手的现实场景：不同仓库独立维护、环境依赖冲突明显、数据与结果约定差异很大。换言之，现有研究证明了统一抽象的重要性，但仍未回答“异构符号回归生态如何在同一平台上公平运行”这一平台层问题。

\subsection{工程化符号回归系统与大语言模型驱动方法}

工程化软件栈的成熟，使符号回归逐渐从研究原型走向可直接使用的科研工具。PySR \citep{cranmer2023pysr} 在性能、可用性与工程可接入性之间取得了较好平衡，推动了符号回归在实际科学计算流程中的应用。与此同时，TPSR \citep{tpsr2023} 以及基于大语言模型的 LLM-SR、DrSR \citep{shojaee2024llm,wang2025drsr} 展示了另一条快速演进的技术路线：大模型不再只是作为公式生成器存在，而是通过代码生成、程序推理、经验检索与多阶段候选筛选，直接进入符号回归方法本体。

这类工作说明，符号回归已经从“单一算法”演化为“算法 + 运行环境 + 提示流程 + 后处理逻辑”的复合系统。其积极意义在于能力边界被大幅拓展，但其副作用也同样明显：不同方法的运行条件差异更大，执行链路更长，结果来源更难解释，真值泄露或协议漂移的风险也更高。也正因为如此，统一评测平台和审查机制的研究不再是外围工程，而是大模型时代符号回归研究必须补上的基础层工作。

\subsection{符号回归 benchmark、数据集与评测协议}

在评测层面，SRBench \citep{lacava2021srbench} 的贡献在于把“统一 benchmark 比较”明确为社区问题，并为当代符号回归方法提供了相对一致的对照框架。此后，关于数据集和协议本身的反思不断加强：SRSD \citep{matsubara2022rethinking} 质疑部分常用数据集并不真正服务于科学发现目标，强调变量范围、公式结构和外推场景的重要性；LLM-SRBench \citep{shojaee2025llmsrbench} 面向大语言模型驱动方法重新讨论数据划分与评测形式；Aldeia 等人 \citep{aldeia2025call} 更进一步提出，下一代符号回归 benchmark 必须把预算、公平性和协议设计当作核心问题来研究。

这些研究说明，社区已经逐步形成一个基本共识：符号回归评测不能只看单一误差，也不能只在单次训练/测试划分上做结论，而应综合考虑验证集、ID/OOD 划分、复杂度、稳定性、运行预算与失败率。但现有工作仍然更多停留在“提出 benchmark 设计原则”层面，缺少与真实异构仓库执行体系紧密结合的平台化落地。也就是说，大家已经越来越清楚“应该如何公平比较”，但“如何让不同方法在现实工程条件下真正被公平比较”仍然没有被彻底解决。

\subsection{可复现机器学习基础设施与研究纪律}

可复现性问题并不仅存在于符号回归，而是整个机器学习研究共同面对的基础问题。\citet{pineau2021improving} 对 NeurIPS 可复现性计划的总结表明，代码可用性、实验细节透明度、评测协议完整性和资源预算说明，已经成为高质量研究的基本要求。对于符号回归而言，这一问题往往更加尖锐，因为其方法常常跨越多语言、多运行环境，并依赖表达式解析、常数优化和自定义后处理等复杂环节，只要其中任意一环缺乏约束，结果就可能因为环境差异或脚本细节而出现明显偏移。

从这一视角看，本文所强调的平台不应被理解为简单的“跑分脚本集合”，而应被理解为一种具备研究纪律的实验基底。它至少需要解决四个问题：第一，方法如何接入并保持接口一致；第二，数据语义如何冻结并可审查；第三，运行产物如何被完整记录；第四，后续自动化改进如何在不误读上下文的前提下消费这些工件。现有可复现性研究为这些问题提供了原则，但尚缺少面向符号回归这一特定任务的系统性实现。

\subsection{自动算法设计、程序搜索与 Agent 化研究}

如果说前述工作主要解决“如何让机器发现公式”，那么近年来另一条快速发展的研究线则开始回答“如何让机器改进发现公式的算法”。AlphaDev \citep{mankowitz2023alphadev} 说明机器可以在程序空间中搜索出具有工程价值的算法结构；FunSearch \citep{romeraparedes2024funsearch} 证明了大语言模型与程序搜索协同可以发现新的数学构造；The AI Scientist 与 AutoSOTA \citep{lu2024aiscientist,li2026autosota} 则进一步把研究对象从单个算法推进到研究流程本身，尝试让系统自动提出方案、生成实验、分析结果并形成后续改进。

这些工作直接启发了本文第二层研究，但它们也留下了一个对本课题至关重要的空白：大多数自动化研究系统默认存在一个稳定、清晰、可执行的实验基底，而符号回归现实生态并不具备这一前提。更进一步说，现有自动化系统更擅长开放式探索，而不是在严格 benchmark 边界内做可审查、可回滚、可复现的受约束改进。对于符号回归这种同时涉及公式搜索、数据语义、科学解释和真值隔离的任务而言，这一空白恰恰决定了自动改进是否具有研究可信度。

\subsection{现有研究的不足与本文切入点}

综合上述文献可以看到，现有研究已经完成了从经典符号回归到深度符号回归、从单方法比较到 benchmark 反思、从工程化工具到 LLM 驱动系统、再到自动化研究代理的连续演进。然而，这些方向之间仍存在明显断裂：方法研究强调能力提升，benchmark 研究强调公平性，基础设施研究强调可复现性，Agent 研究强调自动探索，但很少有工作把这四个目标放在同一研究对象中统一讨论。

因此，本文的切入点并不是简单重复已有某一方向，而是试图完成一次研究对象重组：以符号回归科学发现任务为中心，先建立一个统一、规范、可复现且可审查的评测平台，再在该平台上研究受约束的自动算法改进方法。换言之，本文既不满足于只做 benchmark，也不满足于只做 Agent，而是把“平台支撑改进、改进反哺平台”的闭环作为核心研究问题。

\subsection{研究目标}

本课题的总体目标是：\textbf{构建一个面向科学发现的统一符号回归评测平台，并在此基础上研究审查驱动的自动化算法改进方法，使符号回归算法的比较、复现、诊断与迭代能够在同一可追溯研究基底上完成。}

围绕这一总体目标，本文进一步细化出五个具体目标：
\begin{enumerate}
    \item 建立统一的方法接入与执行接口，使异构符号回归算法能够在同一平台中以一致的方式运行、记录和对比。
    \item 设计规范化数据集模式与协议编译机制，明确训练集、验证集、ID 测试集、OOD 测试集、元数据和真值公式的组织方式。
    \item 构建多维评测体系，不仅报告误差指标，还同时度量表达式复杂度、运行稳定性、失败率、资源开销和协议一致性。
    \item 构建审查模块，将运行日志、候选公式、指标变化和协议风险转化为结构化诊断结果，并作为自动化改进的前置门控。
    \item 设计受约束的 LLM Agent 自动改进方法，使系统能够在不破坏 benchmark 语义的前提下自动提出并验证候选改进动作。
\end{enumerate}

\subsection{拟解决的关键问题}

围绕上述目标，本文拟重点解决以下关键问题：
\begin{enumerate}
    \item \textbf{RQ1：} 如何定义一个足够通用但又不失约束的符号回归平台接口，使传统方法、深度方法和 LLM 驱动方法能够共享统一执行契约？
    \item \textbf{RQ2：} 如何设计数据与协议层的标准化表示，使不同 benchmark 家族在同一平台上可以被公平运行并支持 ID/OOD 评测与真值验证？
    \item \textbf{RQ3：} 如何构建多维评测体系，使评测不再局限于单一 RMSE/NMSE 指标，而能综合表达式复杂度、稳定性、资源开销与复现性？
    \item \textbf{RQ4：} 如何把平台产生的运行日志、错误类型、候选公式、性能曲线和审查结果编码为 LLM Agent 可消费的状态表示？
    \item \textbf{RQ5：} 如何在动作空间、收益函数与协议门控层面约束自动化改进行为，保证其优化过程既有效又不越过科学评测边界？
\end{enumerate}

\subsection{研究假设}

为了使研究问题更加可检验，本文提出如下工作假设：
\begin{itemize}
    \item \textbf{H1：} 统一的平台接口与规范化数据模式能够显著降低符号回归多仓库比较的复现门槛，并提升实验结果的可追溯性。
    \item \textbf{H2：} 相较只看单一误差指标，加入 OOD 指标、复杂度、失败率和协议审查结果的多维评测，更能真实反映方法面向科学发现任务的有效性。
    \item \textbf{H3：} 若自动改进动作受到协议门控、真值隔离和成本约束，则 LLM Agent 可以在相对安全的前提下为符号回归算法带来有效改进。
    \item \textbf{H4：} 平台化评测与自动化改进并不是彼此独立的两个方向，而是能够相互增强：更规范的平台产生更高质量的改进反馈，更成熟的自动改进又能帮助平台暴露其薄弱环节。
\end{itemize}

\section{研究内容}

\subsection{研究内容一：面向科学发现的统一符号回归平台}

本课题的第一项核心内容，是建立面向科学发现的统一符号回归平台。其基本思想不是把所有算法代码改写为同一种实现，而是在保留原始算法特色的前提下，为每个后端定义统一的输入输出契约。平台需至少完成以下功能：算法注册、环境隔离、参数映射、数据读取、训练执行、候选公式提取、预测接口统一、结果归档和错误记录。只有这些基础能力稳定之后，后续评测与自动改进才具备可靠支点。

在当前项目中，这一方向已经具备初步工程基础：项目内已经存在覆盖 QLattice、drsr、dso、e2esr、gplearn、iMCTS、llmsr、pyoperon、pysr、tpsr 等 $10$ 个工具的算法封装目录，并配置了 $8$ 个执行环境。本文将在此基础上进一步系统化平台研究：不是仅仅说明“这些工具能跑”，而是要回答“它们为何可以以统一方式跑”“哪些抽象足以支撑跨方法比较”“哪些运行差异应被平台显式记录而不是被隐藏”。

\subsection{研究内容二：规范化数据集模式与公式感知验证}

评测平台若没有统一数据模式，其统一性就是表面的。为此，本文将研究一套面向符号回归科学发现任务的规范化数据集模式。该模式至少包括 \texttt{train.csv}、\texttt{valid.csv}、\texttt{id\_test.csv}、\texttt{ood\_test.csv} 和 \texttt{metadata.yaml}，并在存在解析真值公式时增加 \texttt{formula.py}。其中，元数据需要明确特征顺序、目标变量、样本数、划分语义、单位说明和可选的任务背景信息；真值公式则不能只是注释性文本，而应能被程序导入并用于数值一致性验证。

这一研究内容的价值有两点。其一，它将 benchmark 的“数据文件集合”提升为“带语义的数据工件”，从而让不同算法在相同语义下消费数据。其二，它为后续的自动化改进提供了明确边界：Agent 可以看到数据划分与任务元信息，但不能直接调用真值公式去作弊，也不能改变 OOD 划分定义来获得虚假提升。通过这种方式，平台不只是统一数据格式，更是在技术上冻结 benchmark 语义。

\subsection{研究内容三：协议编译、运行归档与多维评测}

在统一算法接口与统一数据模式之上，本文将进一步构建协议编译与运行归档层。所谓协议编译，是指在方法、数据集、随机种子、预算、算子集合、超参数模板和评测指标给定后，生成一份可执行、可复现、可归档的实验规格，并将其视为一次实验的“冻结契约”。只有在该契约被固定之后，后续运行结果才具备可比较性。

在评测方面，本文不会只报告单一预测误差，而将设计至少四类指标：一是精度指标，如 RMSE、NMSE；二是泛化指标，如 ID 与 OOD 的差异表现；三是表达式质量指标，如复杂度、长度、常数数量和可化简性；四是运行层指标，如成功率、超时率、平均耗时和资源消耗。通过这种多维评测，平台能够更真实地反映一个算法在科学发现语境下的价值，而不是仅仅用一个最好分数掩盖方法的不稳定性或协议问题。

\subsection{研究内容四：审查优先的诊断与报告机制}

平台的一个关键特性，是在接受任何“算法改进”结论之前先进行审查。本文将研究一套审查优先的诊断机制，其输入包括运行日志、执行状态、指标变化、候选公式、表达式复杂度和数据访问记录，其输出包括失败原因、协议违规风险、改进建议类别和是否允许进入下一轮自动化改进。审查内容包括但不限于：是否修改了评测脚本、是否改变了数据划分、是否使用了真值公式、是否出现了不一致的指标定义、是否存在环境层伪失败、是否只在单次随机种子上偶然提升。

这一设计的理论意义在于，它将“评测”与“改进”之间插入一个明确的合法性层。传统研究中，人类研究者往往凭经验判断某次改动是否合理；在自动化系统中，这一判断必须被形式化为可执行规则。也就是说，本文并不把审查视为附加模块，而是把它作为自动化改进得以安全成立的先验条件。

\subsection{研究内容五：面向符号回归的自动化算法改进方法}

在平台与审查机制稳定之后，本文将研究 LLM Agent 驱动的自动化算法改进方法。这里的“算法改进”并不局限于发明全新算法，更实际的研究路径是把其定义为受约束动作空间中的一系列可执行变换，包括：超参数调整、算子集合裁剪、后处理开关选择、提示词重写、候选重排策略调整、常数优化配置调整、资源预算微调以及环境修复建议等。Agent 的任务不是自由发挥，而是在平台提供的状态表示与规则库中选择高价值、低风险的动作。

本文拟研究的核心问题是：如何把平台状态表示、动作空间、收益函数和审查门控耦合起来，使自动化改进具有可解释性。具体而言，Agent 看到的不应是原始日志文本堆，而应是结构化状态；Agent 输出的不应是模糊建议，而应是可以被执行器落实的受约束动作；Agent 的成功不应只由单次误差下降定义，而应由 OOD 提升、复杂度控制、运行稳定性和协议合法性共同决定。

\subsection{研究内容六：知识归档与人机协同改进闭环}

考虑到硕士阶段研究的实际边界，本文不会追求一个完全自治、无限制迭代的科研代理，而将重点放在“可用的半自动闭环”上。平台会把每轮运行与改进产生的配置、日志、公式、指标、失败模式和审查结论归档为结构化工件，为后续轮次提供上下文。人类研究者可以对 Agent 的建议进行确认、驳回或修正，从而形成“平台执行 - 审查诊断 - Agent 提议 - 人类确认 - 再执行”的稳健闭环。

这种设计兼顾了现实可行性与研究价值。一方面，它避免了当前大模型系统在高风险自主决策上的不稳定性；另一方面，它仍然能够验证第二层研究问题，即机器是否能够在统一平台的边界之内自动改进科学发现工具。

与传统“先设计新算法、再找数据测试”的研究路径相比，本文的路线更强调基础设施先行。原因在于：如果基础平台不稳固，后续任何自动化改进都可能是在不一致的协议上进行，因而难以解释其结果来源。只有当实验输入、运行环境、结果格式和审查规则都被明确固定后，自动改进行为才具有科学研究意义。

\subsection{形式化问题定义}

为便于后续描述，本文对关键对象做如下形式化定义。记符号回归工具集合为 $\mathcal{T}=\{t_1,\dots,t_n\}$，数据集集合为 $\mathcal{D}=\{d_1,\dots,d_m\}$，其中每个数据集都包含训练、验证、ID 测试和 OOD 测试四类划分，即
\[
d = \left(D_{\text{train}}, D_{\text{valid}}, D_{\text{id}}, D_{\text{ood}}, M, F^\star\right),
\]
其中 $M$ 表示元数据，$F^\star$ 表示可选的真值公式工件。一个实验协议可表示为
\[
p = (t, d, \theta, b, s, \mu),
\]
其中 $t \in \mathcal{T}$ 是工具，$\theta$ 是工具配置，$b$ 是计算预算，$s$ 是随机种子，$\mu$ 是评测配置。协议编译器的职责，是把抽象协议 $p$ 转换为平台可执行的冻结工件 $\hat{p}$。

一次运行的结果记为
\[
r = \left(\hat{y}_{\text{id}}, \hat{y}_{\text{ood}}, E, C, L, A\right),
\]
其中 $\hat{y}_{\text{id}}$ 与 $\hat{y}_{\text{ood}}$ 表示预测输出，$E$ 表示误差指标集合，$C$ 表示复杂度指标集合，$L$ 表示日志与执行状态，$A$ 表示审查结果。平台最终面对的并不是单一数值，而是一组结构化工件。

在自动改进层，定义受约束动作空间为 $\mathcal{U}_{\text{safe}}=\{u_1,\dots,u_k\}$，每个动作 $u$ 会把原始协议 $\hat{p}$ 变换为新协议 $\hat{p}'=u(\hat{p})$。但这种变换必须满足协议合法性门控
\[
G(\hat{p}, \hat{p}') = 1,
\]
其中 $G$ 用于检查数据划分、评测脚本、真值可见性和预算上限是否被破坏。只有当门控通过时，动作才允许进入执行阶段。

\subsection{统一方法接入与环境隔离}

统一方法接入的核心，不是强行消灭算法差异，而是抽取最小公共接口。本文拟将工具接入契约定义为：\texttt{fit}、\texttt{predict}、\texttt{get\_optimal\_equation} 和 \texttt{get\_total\_equations} 四个核心能力，再配合运行配置、环境名、超时策略和产物归档结构完成统一执行。对于某些原始仓库并不天然满足这一接口的情况，平台将通过 wrapper 层负责输入转换、脚本适配和结果规整。

环境隔离则用于解决符号回归生态中最常见的依赖冲突问题。当前项目中已配置 $8$ 个受控运行环境，说明多工具共存并不是一个假想问题，而是现实约束。本文将在平台设计中把环境视为协议的一部分，而不是后台实现细节。也就是说，一次实验不仅要记录“跑了哪个工具”，还要记录“在哪个环境中、以何种依赖状态、使用何种运行时参数跑了这个工具”。这种记录方式是后续复现与诊断的基础。

\subsection{规范化数据模式与公式验证机制}

数据模式统一的关键，是将原本散落在各个 benchmark 中的隐含约定显式化。本文拟采用“文件层 + 元数据层 + 公式层”三层组织结构：文件层用于保存不同划分的数据表；元数据层用于记录目标列、特征列、变量顺序、单位说明、任务描述和样本计数；公式层则保存解析真值公式，并提供数值验证入口。这样一来，数据集不再是若干 CSV 文件的集合，而是一个具备可执行语义的实验工件。

对于具有已知公式的数据集，本文拟在平台中引入公式感知验证机制。验证器将导入真值函数，依据元数据中的变量顺序对参数进行映射，并在各个数据划分上计算目标值与真值公式输出的一致性。这个过程的目标不是帮助模型作弊，而是确认数据工件本身是自洽的。若数据文件与公式不一致，则平台应优先报告数据质量问题，而不是直接运行下游算法。这样的设计有助于把数据错误、代码错误和算法错误区分开来。

\subsection{多维评测指标体系}

传统符号回归研究常用 RMSE、MSE 或 $R^2$ 来衡量性能，但这对科学发现任务显然不够。本文计划采用如表~\ref{tab:metric-system} 所示的多维指标体系。对误差类指标，重点报告 ID 与 OOD 两个层面的 RMSE 和 NMSE；对公式类指标，关注表达式长度、树复杂度、常数数量、算子多样性和可化简程度；对运行类指标，记录成功率、平均耗时、峰值资源开销、超时率和崩溃率；对协议类指标，则记录是否存在非法数据访问、脚本改写、评测口径变化或真值泄露风险。

在此基础上，本文进一步定义一个面向自动改进的综合效用函数：
\[
U(u) = \lambda_1 \Delta S_{\text{ood}} + \lambda_2 \Delta S_{\text{id}} - \lambda_3 \Delta C_{\text{comp}} - \lambda_4 \Delta C_{\text{run}} - \lambda_5 R_{\text{audit}},
\]
其中 $\Delta S_{\text{ood}}$ 与 $\Delta S_{\text{id}}$ 表示改动前后在 OOD 与 ID 指标上的收益，$\Delta C_{\text{comp}}$ 表示复杂度代价，$\Delta C_{\text{run}}$ 表示运行成本变化，$R_{\text{audit}}$ 表示协议风险。该定义体现了本文的基本立场：自动化改进不能只追求更低误差，还必须在复杂度、成本和合法性之间做平衡。

\subsection{审查优先的门控机制}

平台的一个关键创新点在于将审查前置。对于任意候选改进动作 $u$，系统并不直接执行，而是先通过门控函数 $G(\hat{p},\hat{p}')$ 检查其是否合法。本文拟将门控规则分为四类：数据规则、脚本规则、资源规则和真值规则。数据规则限制任何动作都不得更改 \texttt{train}/\texttt{valid}/\texttt{id}/\texttt{ood} 的定义；脚本规则限制 Agent 不得修改核心评测脚本和结果汇总逻辑；资源规则限制动作必须在既定预算内运行；真值规则限制任何搜索与提示中都不得使用 \texttt{formula.py} 的信息。

这种设计使得自动改进从一开始就不再是“开放域代码修改”，而是“协议内有边界优化”。在硕士阶段，这一点尤其重要，因为它既降低了系统设计复杂度，也提升了结果解释性。最终，平台希望输出的不只是“Agent 帮助某个工具分数变好了”，而是“Agent 在哪些约束内采用了哪些动作，这些动作为何有效，以及它们是否能在多个任务上稳定复现”。

\subsection{基于 LLM Agent 的自动改进方法}

本文计划采用“状态表示 - 动作生成 - 审查门控 - 执行验证 - 经验归档”的五阶段 Agent 改进框架。首先，状态构造模块将把平台日志、错误码、历史配置、候选公式、指标向量和审查结果编码为结构化上下文；其次，LLM Agent 根据上下文生成若干候选改进动作；然后，由门控模块过滤掉违反协议的动作；之后，对合法动作进行隔离执行并回收结果；最后，将动作效果写入经验库，为下一轮改进提供先验。

在动作空间设计上，本文拟采用分层策略。第一类是\textbf{运行修复动作}，主要解决依赖缺失、超时、路径错误或解析失败等工程问题；第二类是\textbf{配置优化动作}，包括超参数调整、算子集合控制、种群大小或搜索深度变化等；第三类是\textbf{后处理动作}，包括公式化简、常数重拟合、候选重排或表达式筛选；第四类是\textbf{LLM 特有动作}，包括提示词重写、思维链模板替换、示例检索控制和代码生成约束。通过分层动作空间，平台可以更清晰地识别“到底是哪一类修改带来了收益”。

为了避免 Agent 在局部任务上过拟合，本文将优先研究“小步、受约束、可回滚”的改进策略，而不是大幅重写算法主体。也就是说，本文更关注自动化改进的研究方法，而不是追求极端自主性。这种定位更符合开题阶段的可实现路径，也更便于在后续实验中做系统性消融。

\subsection{知识归档与经验复用}

自动改进若想积累能力，就必须有结构化经验存储。本文拟将每次运行与改进的核心信息记录为条目：
\[
e = (\hat{p}, u, r, A, \Delta),
\]
其中 $\hat{p}$ 是原始协议，$u$ 是动作，$r$ 是结果，$A$ 是审查结论，$\Delta$ 是动作前后差异摘要。经验库的作用有三点：一是为 Agent 提供历史成功/失败案例；二是为人工研究者提供可追溯的调试材料；三是为平台研究本身提供元数据，用于分析哪些工具更依赖哪些动作、哪些数据集更容易触发协议风险、哪些错误类型最影响自动改进效率。

\section{实验设计与评测方案}

\subsection{实验目标}

围绕前述研究问题，本文计划通过实验回答三类问题。第一类是平台问题：统一平台是否真的能够承载多种异构符号回归工具，并在相同协议下完成评测。第二类是评测问题：多维指标与审查机制是否能够揭示单一误差指标无法体现的现象，例如 OOD 泛化退化、复杂度膨胀或运行不稳定。第三类是自动改进问题：在受约束动作空间和审查门控下，LLM Agent 是否能够对部分工具或任务产生稳定、可复现的改进收益。

\subsection{实验对象与数据来源}

实验对象将由两部分组成。第一部分是平台当前已具备基础接入的符号回归工具集合，包括 QLattice、drsr、dso、e2esr、gplearn、iMCTS、llmsr、pyoperon、pysr 和 tpsr。第二部分是计划纳入的规范化数据集集合。当前项目内已存在 BPG0、BPG1、BPG2、oscillator1、oscillator2 和 stressstrain 等示例或试点数据资产，这些数据集覆盖了解析真值可验证场景、振子类任务和材料应力应变类任务。后续研究中，本文还将继续吸收 SRBench、SRSD 和 LLM-SRBench 风格数据源，并统一转换到平台规定的目录结构之下。

这里特别强调数据来源的阶段性安排。开题阶段并不要求一次性完成所有数据集接入，而是要先建立数据标准化与验证流程，并在少量高质量样例上验证可行性。等平台与协议稳定之后，再逐步扩展 benchmark 覆盖面。这样做既符合研究节奏，也能避免因为前期数据规模过大而掩盖平台设计问题。

\subsection{对比方案与基线设置}

为了验证自动化改进方法的有效性，本文拟设置以下几类基线：
\begin{enumerate}
    \item \textbf{原始工具基线：} 使用平台默认配置运行各工具，作为零改进基线。
    \item \textbf{人工轻量调参基线：} 由研究者手工做少量超参数调整，比较其与 Agent 改进的差异。
    \item \textbf{无审查 Agent 基线：} 若条件允许，可设置一个不经过完整审查门控的简化 Agent，用于证明审查机制的必要性。
    \item \textbf{分层动作消融基线：} 分别只允许运行修复动作、只允许配置优化动作、只允许后处理动作，评估不同动作层的贡献。
    \item \textbf{单指标优化基线：} 只优化 ID 误差的改进策略，与多目标优化策略比较，观察其在 OOD 和复杂度上的副作用。
\end{enumerate}

\subsection{实验指标与分析维度}

实验分析将围绕五个维度展开。第一，\textbf{性能维度}：主要考察 ID/OOD RMSE、NMSE 和多次运行均值。第二，\textbf{复杂度维度}：考察表达式长度、树深、常数数量与可简化程度。第三，\textbf{稳定性维度}：考察随机种子方差、失败率、超时率以及不同环境下的复现一致性。第四，\textbf{成本维度}：考察运行耗时、调用次数、LLM 预算与总计算开销。第五，\textbf{合法性维度}：考察改进过程中是否触发协议风险、是否存在真值泄露或越界修改。

需要说明的是，本文不会用一个单一综合分数掩盖多维现象。相反，平台的优势恰恰在于保留并展示这些维度之间的张力。例如，某个动作可能显著降低了 ID 误差，但同时导致 OOD 崩溃和复杂度飙升；另一个动作可能仅带来小幅精度提升，却显著降低失败率和运行成本。对于科学发现任务而言，后者并不一定比前者更差，甚至可能更有研究价值。

\subsection{核心实验设计}

本文计划设置以下核心实验：
\begin{enumerate}
    \item \textbf{E1：平台覆盖实验。} 统计有多少异构工具能够在统一平台内成功运行，并分析失败类型分布。
    \item \textbf{E2：数据模式验证实验。} 对存在真值公式的数据集执行数值一致性校验，验证规范化数据工件的正确性。
    \item \textbf{E3：多维评测对比实验。} 比较仅看单一误差指标与引入 OOD、复杂度、运行指标后的结论差异。
    \item \textbf{E4：Agent 自动改进有效性实验。} 检验 Agent 在合法动作空间内能否带来稳定收益。
    \item \textbf{E5：审查机制必要性实验。} 分析没有审查门控时，自动改进是否更容易出现不公平提升或不可解释结果。
    \item \textbf{E6：动作空间消融实验。} 识别运行修复、配置优化、后处理和提示词调整各自的贡献。
    \item \textbf{E7：跨任务迁移实验。} 研究某些经验是否能从一个数据集迁移到另一类任务。
    \item \textbf{E8：案例研究。} 选取 1--2 个工具做完整的改进链路分析，展示平台如何支持从失败诊断到动作验证的全过程。
\end{enumerate}

\subsection{公平性与复现性控制}

为了保证实验结论可信，本文将把公平性控制作为实验设计的硬约束。具体来说：第一，固定每类实验的预算上限，包括搜索轮数、训练时间、LLM 调用次数和随机种子数；第二，统一结果记录格式，避免不同工具采用不同汇总逻辑；第三，对关键实验至少进行多次重复运行，报告均值和方差；第四，对所有 Agent 生成动作做审计记录，必要时保留 diff 级证据；第五，将平台配置、环境信息、数据版本与脚本版本纳入归档工件。这样的设计使得实验不仅能比较方法，也能接受后续研究者复查。

\subsection{预期结果}

预计通过上述实验，本文将得到以下几类结果。首先，能够证明统一平台对于异构 SR 工具接入与对比具有实际价值，并能显著提升实验复现性。其次，能够证明多维评测比单一误差指标更适合衡量科学发现型符号回归任务，尤其是在 OOD 场景和公式复杂度控制方面。再次，预计可以在部分工具与任务上验证审查驱动 Agent 改进的有效性，即使其收益不一定表现为“大幅刷新榜单”，也能够在降低失败率、提升 OOD 表现或改进表达式质量等方面体现价值。最后，通过失败案例分析，本文还能够总结自动化改进在符号回归场景下的边界条件，这对于后续研究同样重要。

\section{项目基础与可行性分析}

\subsection{已有工程基础}

本课题并不是从零开始。当前项目仓库已经具备较为明确的平台雏形：算法目录中已存在 $10$ 个符号回归工具封装；配置中已有 $8$ 个执行环境；\texttt{check} 目录下已有 $10$ 个与具体工具对应的验收脚本；\texttt{examples} 目录下已经存在 BPG0、BPG1、BPG2、oscillator1、oscillator2、stressstrain 等数据资产；\texttt{paper} 目录下还已有中英文论文草稿与 benchmark 图表。这些现有资产意味着本课题可以直接从“已有原型的系统化研究”出发，而不是停留在概念想象层面。

尤其重要的是，现有项目已经暴露出真实问题。例如，不同工具对环境隔离的需求不同；部分数据集同时存在 \texttt{id\_test.csv} 与历史命名的 \texttt{test\_id.csv}；不同后端的结果格式并不完全一致；个别工具存在运行时不稳定、超时或依赖不兼容等现象。这些问题恰恰构成了本课题的研究素材。换言之，平台研究最怕的是“问题都不存在”，而本项目当前状态已经充分说明问题真实存在且具备研究价值。

\subsection{理论可行性}

从理论上看，本课题所依赖的三个知识基础都已经较为成熟。第一，符号回归本身已经形成较丰富的方法谱系和 benchmark 讨论，可为平台层与评测层提供足够的研究背景。第二，面向科研与程序优化的大语言模型 Agent 已经有一系列代表性工作，说明“自动提出并验证改进动作”并非不可行。第三，软件工程和可复现机器学习的相关原则为平台设计、协议管理和结果归档提供了成熟思路。因此，本文并不是试图同时发明多个完全未知的问题，而是在已有研究基础上做针对性整合与推进。

\subsection{工程可行性}

从工程实现看，本课题的难点主要在系统整合而非底层技术不可达。当前仓库已经实现基本的 wrapper 结构、环境管理器、配置管理器和部分 benchmark 运行逻辑；本地环境也已准备好 PySR/Julia 缓存路径，可减少某些关键后端的环境构建成本。此外，研究目标本身也具有分阶段落地特性：可以先完成平台统一和数据规范化，再逐步增加审查机制和自动改进模块，而不需要一步到位实现完整自治系统。

\subsection{实验可行性}

从实验资源看，本课题并不要求一开始就在超大规模 benchmark 上进行重计算。相反，本文强调先用少量高质量、语义清晰的数据集和代表性工具做验证，再按需扩展覆盖面。这样既能控制实验成本，也有助于更精细地观察自动改进机制是否有效。对于需要调用大语言模型的环节，也可以通过限制动作空间、控制调用轮数和缓存上下文来保持成本可控。

\subsection{风险与应对思路}

本课题的主要风险并不在“想法过于空泛”，而在“工程整合复杂、实验路径容易发散”。为此，本文将坚持以下原则：一是平台先行，先把运行和记录打牢，再引入 Agent；二是动作受限，不做无限开放的代码修改；三是优先分析失败案例，而不是只追逐分数；四是把每个阶段都设计为可验收的小闭环。这样的研究策略更符合硕士阶段项目管理要求，也更容易形成阶段性产出。

\section{研究难点与拟突破点}

\subsection{难点一：异构符号回归仓库的统一承载}

符号回归平台研究最大的现实难点，在于现有方法生态并不是围绕“统一接入”而设计的。不同仓库在编程语言、第三方依赖、文件读取方式、参数命名、输出格式、随机种子控制、失败处理和候选公式表示方面都存在明显差异。许多算法甚至默认自己是实验流程中的唯一主角，而非一个可被纳入共享评测体系的组件。因此，把它们放进同一平台中并不是“写几个 adapter”就能解决的问题，而需要回答哪些差异应该被保留，哪些差异必须被规整，哪些差异需要被显式记录。

针对这一难点，本文拟突破的方向不是追求百分之百接口同构，而是构建一套“最小公共契约 + 差异外显记录”的方法。一方面，平台只强制要求核心能力对齐，例如训练、预测、最优公式输出和候选公式获取；另一方面，保留各工具的扩展字段，并把这些字段纳入结果归档。这样既避免了为了统一而牺牲方法特色，也避免了因差异被隐藏而导致结果不可解释。

\subsection{难点二：评测协议冻结与 benchmark 语义保持}

第二个关键难点，是如何在平台和 Agent 共同参与的场景中冻结 benchmark 语义。传统人工实验中，研究者可能凭经验避免不公平操作；但在自动化改进中，如果没有技术性边界，系统可能很容易通过改变数据划分、删改失败样本、修改指标计算方式、绕过 OOD 评测或者隐式使用真值公式来获得“伪提升”。这类问题一旦出现，整个平台的研究价值就会被削弱。

本文拟突破的关键点，是把 benchmark 语义显式编码到协议工件中。数据划分、脚本版本、指标定义、预算上限和真值可见性都将被写入冻结协议，并由审查模块强制检查。这样，平台不会只是“跑实验”，而是“运行一份被冻结且可检查的科学比较契约”。这一突破点对于本文第二层研究尤其重要，因为只有在协议被冻结后，自动化改进的收益才具有学术可信度。

\subsection{难点三：让 LLM Agent 真正服务于符号回归而不是扰动平台}

当前大语言模型 Agent 的一个普遍问题，是生成能力很强，但任务边界意识较弱。在符号回归场景中，这会带来两个风险：其一，Agent 可能提出大量表面合理、实际不可执行的动作；其二，Agent 可能试图通过非本质路径提升结果，例如直接利用真值信息、绕过评测流程或引入额外外部知识，从而偏离研究目标。若不解决这一问题，所谓“自动化算法改进”很可能沦为不稳定的脚本扰动。

为此，本文拟突破的重点是把 Agent 的自由文本建议收束为结构化、模板化、可校验的动作空间。Agent 不直接修改整套实验流程，而是基于平台提供的动作模板做选择与参数化填写。这样一来，LLM 的优势体现在综合判断、经验迁移和文本推理上，而执行和审查仍由平台掌控。这种“LLM 负责提出候选，平台负责落地和裁决”的设计，是本文拟解决的核心工程方法问题。

\subsection{难点四：自动改进收益的稳定性与可迁移性}

即便 Agent 能在某一次实验中带来性能改善，也不意味着它具有研究价值。硕士阶段论文更需要回答的是：改进是否稳定、是否可复现、是否只对单个任务有效、是否会在另一个任务上带来副作用。因此，本文的难点不仅是“能不能改好”，更是“改好的结论是否能够经得起跨任务和跨运行验证”。

对应地，本文拟突破的第四个点，是把自动改进研究从“单次成功案例展示”提升为“多任务、多轮次、带失败分析的稳定性研究”。也就是说，本文不会把某次偶然提升视为充分结果，而将通过重复实验、动作消融、跨任务迁移分析和失败案例归档来证明改进机制本身的规律性。只有这样，第二层研究才能真正形成方法论贡献。

\section{预期创新点}

结合前文研究内容，本文预期可能形成以下创新点：
\begin{enumerate}
    \item \textbf{研究对象重组创新。} 本文不把“符号回归评测平台”和“自动化算法改进”视为两个彼此独立的问题，而是提出一个双层研究框架：第一层研究机器如何面向科学发现恢复公式，第二层研究机器如何在统一平台上自动改进恢复公式的工具。这一重组使平台、协议和 Agent 能够在同一研究对象中形成闭环。
    \item \textbf{面向科学发现的平台化评测创新。} 相比仅提供算法封装或 benchmark 列表，本文拟提出一套覆盖工具接入、环境隔离、数据语义冻结、协议编译、结果归档和多维评测的统一平台方案，使“公平比较”从原则层落到可执行层。
    \item \textbf{公式感知数据工件与协议边界创新。} 本文拟把数据集从普通文件集合提升为包含 train/valid/ID/OOD 划分、元数据与可执行真值公式的结构化工件，并把真值隔离、脚本版本和预算约束写入冻结协议，为后续评测与自动改进提供清晰边界。
    \item \textbf{审查优先的自动改进方法创新。} 针对现有 Agent 系统边界意识不足的问题，本文拟提出“先审查、后改进”的自动化闭环，通过状态表示、动作模板和协议门控约束 LLM Agent 的行为，使自动改进从开放式探索转化为可审查、可回滚、可复现的受约束优化。
    \item \textbf{面向科学发现任务的多目标改进建模创新。} 本文不以单一误差下降定义改进成败，而是将 OOD 表现、表达式复杂度、运行成本、稳定性与协议风险共同纳入收益函数，从而使自动化改进的目标更贴近科学发现任务的真实需求，也更便于后续实验分析与方法比较。
\end{enumerate}

需要说明的是，本文的创新预期并不主要体现在重新发明一个底层符号回归模型，而更可能体现在研究问题重组、平台抽象、协议设计以及自动化边界建模上。对硕士阶段研究而言，这类创新更具可验证性：它们既能够落到系统实现和实验方案中，又能在论文中通过平台能力、评测结果和案例分析被清晰证明。

\section{预期成果}

本课题预期形成以下几类成果：
\begin{itemize}
    \item \textbf{平台成果：} 一个面向科学发现的符号回归统一评测平台，具备多工具接入、数据标准化、协议编译、结果归档和多维评测能力。
    \item \textbf{方法成果：} 一套审查驱动的自动化算法改进方法，包括状态表示、动作空间、多目标收益函数和门控机制。
    \item \textbf{实验成果：} 一组覆盖典型工具与典型数据集的 benchmark 结果、消融实验与案例分析。
    \item \textbf{文档成果：} 开题报告、中期报告、论文初稿及最终学位论文，必要时配套技术文档和使用说明。
    \item \textbf{可复用工件：} 规范化数据集模板、工具接入脚手架、审查报告模板和经验归档格式。
\end{itemize}

从更长期的角度看，这些成果还可以为后续扩展提供基础，例如接入更多科学发现任务、拓展到 PDE 发现或系统辨识、引入更强的实验规划 Agent 等。

\section{结论}

本文围绕《面向科学发现的符号回归评测平台与自动化算法改进方法研究》这一选题，提出了一个以双层结构为核心的研究框架。第一层关注如何让机器更稳定、更公平、更可复现地承担“从数据到公式”的科学发现任务；第二层关注如何让机器进一步承担“改进发现公式工具”的任务。在这一框架下，统一平台并不是附属工程，而是自动化改进得以成立的研究基底；审查机制也不是额外负担，而是保障自动化系统不越界的核心条件。

结合当前项目已经具备的工程基础，本文认为该课题具有明确的问题价值、较强的技术可行性和较清晰的阶段推进路径。预计通过本课题研究，可以为符号回归科学发现任务提供一个更规范、更可复现、更易扩展的研究基础设施，同时为 AI for AI4Science 场景下的自动算法改进方法提供一条可操作的实现路径。

\section{附录G：典型自动改进案例流程模板}

为了使第二层研究在后续实现中更具可操作性，本文在开题阶段就给出一个典型自动改进案例流程模板。该模板并不是最终算法，而是用于约束实验组织与结果汇报格式。

\begin{enumerate}
    \item 选择一个具备代表性的工具-数据集组合，例如 \texttt{pysr} 在 \texttt{oscillator2} 上的运行任务，并固定预算、随机种子和评测协议。
    \item 运行平台基线，收集日志、指标、候选公式、失败信息和资源消耗，生成首轮归档工件。
    \item 由审查模块读取首轮工件，判断该任务当前的主要瓶颈属于运行失败、配置不稳、公式复杂度过高还是 OOD 退化。
    \item 状态构造器将审查结论、历史相似案例、工具配置和结果摘要编码为 Agent 输入。
    \item LLM Agent 在动作模板集合中生成若干候选动作，例如缩减算子集合、调整搜索温度、修改提示模板或开启后处理化简。
    \item 门控模块逐条检查这些动作是否违反数据、脚本、预算和真值可见性约束，过滤掉非法动作。
    \item 平台执行剩余合法动作，并对每个动作生成新的运行归档、指标报告与风险标签。
    \item 比较动作前后的 OOD 性能、复杂度、成本和审查结果，若收益明显且风险可接受，则记录为成功案例；否则触发回滚或停机。
    \item 将成功或失败经验写入经验库，形成下轮改进的可检索上下文。
\end{enumerate}

这个模板的意义在于，它使后续论文中的案例研究能够遵循统一叙事结构，即“问题出现在哪里、Agent 看到了什么、提出了什么、平台如何审查、最终结果如何变化”。这比简单展示“某次改了参数后分数更好”更有研究价值，也更符合本文强调的可追溯自动化研究路线。
