# 工程架构与实验基础设施

## 你的工程思路经历了一次关键转向

最开始你已经意识到，不能直接在 wrapper 层硬封装所有算法。  
问题在于这样会导致：

- 耦合太高
- 难以恢复实验状态
- 不利于不同算法的差异化处理
- 后期维护成本很大

所以你后来的方向转成了：

> 先精简原始算法代码，再在统一实验目录之上做抽象

## 核心架构：实验目录是第一公民

你逐步形成的架构核心是：

- 一个问题对应一个实验目录
- 所有输入、过程状态、输出结果都在这个目录里
- 支持写盘、读盘、恢复、二次预测
- 目录组织优先考虑“数据集 + 算法”

这个设计很重要，因为它把一次实验从“临时运行的脚本”变成了“可管理对象”。

## 这个架构天然支持什么能力？

### 1. 结果持久化
- 中间过程可落盘
- 最终结果可直接归档

### 2. 实验恢复
- 可以重新找到实验上下文
- 有机会支持 resume / replay

### 3. 多算法横向比较
- 同一数据集下不同算法结果更容易对齐
- 更符合 benchmark 的科研视角

### 4. 后续服务化
- 后面如果要接前端、知识空间、leaderboard，这个目录结构是很好的基础

## 你的系统工程视角已经很强了

从后续文档看，你已经在处理：

- Redis
- Celery
- Flower
- Worker 部署
- 反向 SSH 隧道
- 结果收集
- 中心节点与分布式节点迁移
- local WandB

这意味着你的 benchmark 已经不是“代码工程”，而是：

> 一个带有调度、监控、容错和汇总能力的实验基础设施系统

## 当前基础设施的标准运行链路

1. 启动 Redis
2. 启动 Flower
3. 部署 Worker
4. 提交探针任务
5. 实时监控任务
6. 收集结果并做难度分析

## 可以继续补强的点

### 1. 统一输出 schema
每个算法最终都应该输出到同一种结果格式里，例如：
- 公式
- 参数
- 训练耗时
- 迭代数
- 最终指标
- 是否恢复成功

### 2. 统一 resume 机制
不同算法恢复逻辑不同，但 benchmark 层最好仍然抽象出统一恢复接口。

### 3. 实验元数据登记
建议把以下内容固定写入 metadata：
- 数据集 ID
- 算法名
- seed
- prompt / hyperparameter
- 运行节点
- 开始 / 结束时间
- 失败原因

## 这一条线的意义

你这几个月已经逐渐认识到：

> benchmark 的瓶颈不仅在算法，更在实验基础设施。

这是非常重要的成熟标志。
