# 待解决问题与论文叙事

## 你已经开始主动思考“别人会怎么质疑你”

从 TODO 和自问自答里可以看出来，你已经不只是往前做，而是在提前构建：

- 审稿人可能会问什么
- 复现者可能会卡在哪里
- 你的结果哪里最容易被质疑
- 论文里哪些地方必须说清楚

这是很成熟的研究推进方式。

## 目前最关键的几个风险点

### 1. 基线复现问题
如果你的 LLM-SR 或其他基线结果明显差于原论文，别人一定会问：
- 是不是你复现错了？
- 参数对齐了吗？
- seed 对齐了吗？
- split 对齐了吗？
- 模型 backbone 对齐了吗？

这类问题需要提前准备一整套回答框架。

### 2. ID / OOD 拆分不清
如果：
- 百分位数没有明确
- 划分规则不透明
- 参数拟合的数据范围说不清

那么整个 OOD 结果都会失去说服力。

### 3. 指标不够贴近 SR 本质
如果继续使用较旧或解释性弱的指标，而不补充结构类指标，容易被质疑：
- 你到底在比较拟合，还是比较表达式恢复？
- 这个 benchmark 对“公式发现”本质的刻画够不够？

### 4. 算法分类可能被质疑
你已经预判到了：
- uDSR
- TPSR
- RAG-SR

这些都可能被认为不是“纯类别”方法。

所以后面论文里要明确：
- 分类依据是核心创新点，而不是是否使用过多种技术

## 你的论文叙事已经具备雏形

### 可行的主叙事
1. 当前 SR benchmark 存在饱和与混乱问题
2. 现有数据集和接口不足以公平比较新旧方法
3. 我们构建统一基础设施并组织 800 个数据集
4. 我们提出分层评测漏斗与核心子集
5. 我们在 10 个代表性算法上进行了大规模系统实验
6. 除了排行榜，我们还分析了不同方法的适用边界与失效模式

## 适合写进论文的亮点

### 1. 不是单纯集成，而是标准化基础设施
强调：
- 统一接口
- 统一问题表示
- 统一实验目录
- 统一结果收集

### 2. 不是简单挑几个测试集，而是有方法论地构建评测漏斗
强调：
- SR-Cal-25
- SSR-50
- 非重叠设计
- 探针驱动的难度识别

### 3. 不是只给结果表，而是给方法边界分析
强调：
- 不同范式在不同挑战轴上的表现差异
- LLM 方法的有效性边界
- 噪声、维度、复杂度对方法的影响

## 当前最值得优先补的三件事

1. 写清楚 baseline reproduction protocol  
2. 写清楚 ID / OOD split definition  
3. 写清楚指标表与结果表口径

这三件事一旦固定下来，整篇论文的可信度会大幅提升。
